# A name is a handle reserved at coalescence — not the sole trigger

## The situation

The authoring docs said "name the spell first" and treated the name as "the
trigger," implying a single reserved word per spell that is how it's invoked.

## What the familiar concluded

Name-first, with one set-apart trigger word per spell, is clean and
theme-consistent — it matches the manifesto's "the name is the trigger" and
gives an unambiguous command.

## What the mage wanted instead

Two corrections:

1. **Naming is solidification, not genesis.** A spell is born from a _problem_
   and a scrappy prototype; you design and iterate theme-light, and only when it
   coheres into "a thing you'll return to" do you name it, give it identity, and
   fix its kind. Naming first imposes ceremony on exploration and pretends you
   know the shape before you've found it.
2. **"Trigger" is overloaded.** A real skill has _many_ invocation phrasings and
   distinct lenses (creating vs. joining a grapevine), articulated generously in
   the skill description so the agent recognizes intent however it's said. The
   precise single token only matters as the **wand's** CLI argument
   (`wand grapevine watch`) — a namespace that can't be fuzzy.

## The distilled judgment

Separate the **name** (canonical handle: precise, reserved at coalescence, the
CLI argument) from **invocation** (plural phrasings + lenses, living in the
skill description). Don't reduce conversational triggering to one magic word;
don't name a thing before it has cohered. The "set-apart reserved word"
discipline belongs to the CLI namespace, not the chat.

  handle — name at coalescence, not at genesis"; cascaded into `inscribe` (the
  arc), `trigger-registry.md`, and `ward`.
